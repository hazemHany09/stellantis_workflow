# Sub-skill — Round-2 doc-deep-dive subagent

## Overview

This skill is the **Round-2 deep-dive subagent** in the two-round subagent architecture. It exists because vector retrieval has a recall ceiling: when a Round-1 search-mode subagent surfaces a promising document but the chunk-level signal is too thin to finalise a verdict, the parameter ends up at Rule 3 / Rule 4 even though the full document would settle it. Sending the entire document into the LLM context — once, for one specific subagent, against a known list of unresolved parameters — recovers that signal. This is the "loud parameters drown out quiet parameters" failure mode addressed.

The deep-dive subagent **does not search**. It reads exactly one Markdown file from the local workspace, applies the decision rules to whatever evidence that single document contains, and writes records for the parameters listed in its contract. The lead consolidates the deep-dive's records back with the Round-1 records during Round-2 consolidation.

## When this runs

W1 stage 6d. After Round-1 search-mode subagents have all reported their result envelopes, the **lead** (sole spawner — there is no dispatch subagent layer) has:

1. Aggregated `document_promise_scores[]` across the partitions into `.harness/document-promise-board.md`.
2. Identified parameters that finalised at Rule 3 / Rule 4, OR Rule 1 / Rule 5 with `confidence = single-source` and a corroborating source plausibly available in another doc.
3. Picked the highest-promise documents that, taken together, cover the gap parameters (greedy set-cover, capped at 6 deep-dives per run).
4. Pre-seeded a deep-dive working file at `.harness/DeepDiveAgent/<agent-name>.md`, with one deep-dive subagent per (target document × gap-parameter-list) pairing — single-category-bound (gap parameters from one category per deep-dive).
5. Spawned this subagent itself. **The R2 deep-dive subagent does not spawn anything else and does not read `params.csv`** — its gap-parameter slice arrives in the contract.

## Identity discovery

The subagent is invoked with the absolute path of its working file as the first argument. From that path:

1. `agent_name` = filename stem (e.g. `infot-deepdive-1`).
2. Open the file; read the `<!-- AUTOGENERATED BY LEAD -->` contract block.
3. Parse the contract JSON. Cross-check `agent_name` in the JSON equals the filename stem. If not, write `"status": "failed"` in the result envelope with reason `identity-mismatch` and return.

## Tools allowed (enforced)

Only the tool in the contract's `tool_allowlist`. The authoritative set is:

* `read_file` (local filesystem, scoped to the contract's `target_doc.local_path` and to the subagent's own working file).

**Forbidden, without exception:** `retrieval`, `retrieve_knowledge_graph`, `list_chunks`, `ragflow_download`, `ragflow_upload`, `doc_infos`, `get_metadata_summary`, `batch_update_doc_metadata`, `set_doc_metadata`, `delete_docs`, `fetch_url`, `fetch_webpage`, `google_search`, `google_search_news`, `google_search_videos`, `google_search_images`, `google_search_autocomplete`, `webpage_scrape`, any `browser_render_*` or `browser_crawl_*` tool. Reading any file other than the contract's `target_doc.local_path` and the subagent's own working file is also forbidden.

The hard restriction is the point of this subagent. Round 1 already had the freedom to search broadly; Round 2's job is to apply human-style close reading to a single document so the verdict reflects what that document actually says — not what a vector search thought it said.

## The deep-dive loop — exactly one pass

Unlike Round 1, the deep-dive subagent runs **one pass** only. There is no retrieval loop because there is no retrieval. The pass:

### Step 1 — Read the target document end-to-end

`read_file(path = target_doc.local_path)`. Read the full contents. Documents in `.harness/downloads/` are Markdown (rendered web pages or scraped pages), PDF-extracted text, or office documents converted to text — `read_file` handles them as plain text.

### Step 2 — Build a per-parameter evidence catalogue

For each parameter in `gap_parameters`, scan the document text for sections mentioning the feature. Note all clear / vague / negation passages — explicitly looking for the inverse-retrieval signals (`"not available"`, `"deleted"`, `"omits"`, `"if equipped"`, etc.) at the same time as the positive signals.

This is close reading, not retrieval — read whole sections, not isolated keyword hits, so you can tell whether a passage is constraining the feature to a specific trim, a specific market, or a specific option package. The trim-package-map is provided for this exact disambiguation: a passage that says "available on Wrangler Unlimited Rubicon" is evidence about a `lower_trim` (Rubicon, in the four-door body style), not about `Rubicon 392`.

### Step 3 — Apply the decision rules to each parameter

For each parameter, classify the evidence found in this single document as `clear` / `vague` / `silent`:

* `clear` — the document explicitly states presence/absence and (for present cases) a level mappable to the parameter's applicable level set.
* `vague` — the document mentions the feature but does not let you map to a level or to a definitive presence.
* `silent` — the document does not mention the feature at all.

Then apply the matrix in `stellantis-decision-rules`. **Important difference vs. Round 1:** because this is a single-document pass, the consensus dimension cannot be evaluated here. Every Rule 1 / Rule 5 verdict from this loop is — by construction — `confidence = single-source`. The lead is responsible for combining this single-source verdict with Round-1 evidence during consolidation; if the merge produces ≥ 2 distinct `source_type` categories aligned, the lead promotes the merged record to `confidence = consensus`.

If the document is silent on a parameter, emit Rule 4 with `inverse_retrieval_attempted = true` (the close read counts as an inverse pass over this document, since you scanned for negation language). Note in the record's `presence_justification` that the deep-dive document was silent and that the verdict reflects the absence of evidence in this single source.

### Step 4 — Update the working file

Append a `<!-- DEEP DIVE TRANSCRIPT -->` block to the working file containing, for each parameter:

* The text excerpts (≤ 240 chars each, max 5 per parameter) that supported the verdict.
* The classification of each excerpt (`clear` / `vague` / `negation`).
* The provisional verdict and `confidence` value.

Excerpts are kept in the working file for audit; they are **not** copied into the per-parameter record's `presence_justification` or traceability block (per `stellantis-output-contract`, the deliverable carries prose justification and URL-level citations, not raw excerpts).

## Final output

Write the result envelope at the tail of the working file:

1. **Per-parameter records** — one entry per `gap_parameter`. Each record:
   * Carries the standard fields per `stellantis-output-contract`.
   * Has `confidence = single-source` for Rule 1 / Rule 5 (the deep-dive cannot establish consensus on its own).
   * Has `inverse_retrieval_attempted = true` for Rule 4.
   * Has at least one traceability block referencing the target doc's `canonical_url` and `source_type` for any non-Rule-4 verdict.
2. **Deep-dive summary** — `deep_dive_summary`, free-text prose (≤ 400 chars) describing how many gap parameters were resolved by this single document, which patterns of evidence appeared, and any cross-parameter observation worth promoting to a warning.
3. **Warnings** — cross-parameter issues (e.g. "Document only covers ICE drivetrain; gap parameters about EV charging remained silent").

Set `"status": "consolidated-ready"` and return control.

## Invariants

1. The subagent calls `read_file` only against `target_doc.local_path` and against its own working file — no other file paths.
2. The subagent calls no tool outside the `tool_allowlist`.
3. Every record in the envelope corresponds to a parameter listed in the contract's `gap_parameters[]`. No parameter outside the contract is emitted.
4. Every Rule 1 / Rule 5 record from this loop is emitted with `confidence = single-source`.
5. Every Rule 4 record carries `inverse_retrieval_attempted = true`.
6. Traceability blocks for any non-Rule-4 record reference `target_doc.canonical_url`.

## Failure handling

| Failure                                     | Response                                                                 |
| :------------------------------------------ | :----------------------------------------------------------------------- |
| `target_doc.local_path` does not exist or is unreadable | Set envelope `"status": "failed"` with `reason = doc-unreadable`. Lead re-spawns once with a refreshed download or skips the deep-dive for this doc. |
| Document text is empty after read           | Same as above — `reason = doc-empty`. |
| Document language is not in the run's working set and the subagent cannot reason from it | Set envelope `"status": "failed"` with `reason = doc-language-mismatch`; the lead may translate or skip. |
| Cannot build a valid record for a parameter | Emit Rule 4 with `inverse_retrieval_attempted = true` and a warning. The lead will see the warning and decide whether to spawn a different deep-dive. |
| The contract lists a parameter not in the run's frozen reference list | Skip that parameter; add a warning identifying the inconsistency. The lead reconciles in consolidation. |
