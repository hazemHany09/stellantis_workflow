# Sub-skill ŌĆö Subagent classification loop

## Overview

Reads the lead-authored contract from the pre-seeded working file, runs up to 3 main retrieval iterations plus a 3-iteration fallback, applies the six decision rules, and writes a complete result envelope. KB retrieval tools only ŌĆö web search and browser rendering are forbidden.

## When this runs

Whenever the lead spawns a subagent with a pre-seeded working file at `runs/<run-id>/.harness/SubAgent/<agent-name>.md`.

## Identity discovery

The subagent is invoked with the absolute path of its working file as the first argument. From that path:

1. `agent_name` = filename stem (e.g. `ev-1`).
2. Open the file; read the `<!-- AUTOGENERATED BY LEAD -->` contract block.
3. Parse the contract JSON. Cross-check `agent_name` in the JSON equals the filename stem. If not, write `"status": "failed"` in the result envelope with reason `identity-mismatch` and return.

## Tools allowed (enforced)

Only the tools in the contract's `tool_allowlist`. The authoritative set is:

* `retrieval`, `retrieve_knowledge_graph`
* `list_chunks`, `ragflow_download`
* `doc_infos`, `get_metadata_summary`

The subagent never writes document metadata directly. It **stages** metadata patches inside its result envelope; the lead applies them via `batch_update_doc_metadata` during consolidation.

**Forbidden:** `google_search`, `google_search_news`, `google_search_images`, `google_search_videos`, `google_search_autocomplete`, any `browser_render_*` or `browser_crawl_*` tool, `fetch_url`, `create_dataset`, `delete_datasets`, `ragflow_upload`, `upload_with_metadata`, `delete_docs`, `set_doc_metadata`, `batch_update_doc_metadata`. Also forbidden: reading any file under `.harness/Category/`, other `.harness/SubAgent/` files, or the main `STATE.md`.

## Main loop ŌĆö up to 3 iterations (budget from contract)

For each iteration, process parameters that do not yet have a finalised verdict.

### Step 1 ŌĆö Document relevance scan

Use `retrieval` on the KB dataset to identify the most relevant documents for the parameters in this partition. One call per parameter OR one multi-query call per batch of similar parameters. Retain the top-k (default 6) `doc_id`s.

Alternative: call `retrieve_knowledge_graph` when parameters cross-reference entities (rare; use only when `retrieval` alone is ambiguous).

### Step 2 ŌĆö Read

For each shortlisted `doc_id`:

1. Call `list_chunks(doc_id)` to enumerate chunks.
2. To read document content: first check if `.harness/downloads/<doc_id>` (or a previously saved filename for this doc) already exists ŌĆö reuse it if so (cache hit). Otherwise call `ragflow_download(doc_ids=[doc_id], output_dir=".harness/downloads/")`, which saves the file to disk and returns the path. Read only the chunks that match the parameter's description. Alternatively, use the top-ranked chunks returned directly by `retrieval` output when they contain sufficient text.
3. Skim. Classify each source per the decision-rule taxonomy:
   * `silent` ŌĆö parameter not mentioned at all. Discard.
   * `vague` ŌĆö parameter mentioned but level not clearly mappable.
   * `clear` ŌĆö parameter mentioned with level OR explicit presence/absence.

### Step 3 ŌĆö Update working file

Append under the current iteration heading of the working file (see template):

* The retrieval queries issued.
* The doc ids touched + chunk ids downloaded.
* Per parameter: observed evidence snippets, kind (`clear` / `vague`), any newly-added traceability block.

### Step 4 ŌĆö Tentative verdict per parameter

Apply the decision rules:

| Evidence situation                                                                   | Rule    | Output triple                                     |
| :----------------------------------------------------------------------------------- | :------ | :------------------------------------------------ |
| Ōēź1 clear, all clear agree on a valid level                                           | Rule-1  | `(Yes, Success, <level>)`                         |
| All clear agree feature is absent                                                    | Rule-5  | `(No, Success, Empty)`                            |
| Ōēź2 clear sources agree feature is present but disagree on level                      | Rule-2a | `(Yes, Conflict, Empty)`                          |
| Ōēź2 clear sources disagree on presence vs absence                                     | Rule-2b | `(Disputed, Conflict, Empty)`                     |
| Only vague active sources (no clear)                                                 | Rule-3  | `(Yes, Unable to Decide, Empty)`                  |
| No active sources at all (silent-all, all reviewed sources silent on this parameter) | Rule-4  | `(No Information Found, Unable to Decide, Empty)` |

Edge cases:

* Evidence only points at a level **not defined in the parameter's schema_type** ŌĆö this evidence does not map to a level. Treat the source as if it evidenced an undefined tier: it does not produce a `clear` stance. Record an `advisory` of kind `undefined_tier` with the parameter id and doc id.
* **Mixed source set** ŌĆö one clear source supports a valid level; another clear source supports an undefined tier. Disregard the undefined-tier source for classification. The parameter's verdict is decided by the remaining clear source(s). Record the advisory.

A parameter is **finalised** for this loop if the tentative verdict is Rule-1, Rule-2a, Rule-2b, or Rule-5. Rule-3 and Rule-4 are candidates for the fallback loop.

### Loop exit

Stop the main loop when either:

* Every parameter in the partition has a finalised verdict, OR
* The iteration budget (max 3) has been consumed.

## Fallback loop ŌĆö up to 3 iterations

Runs only if any parameters are still at Rule-3 or Rule-4 after the main loop.

### Step 1 ŌĆö Targeted hybrid retrieval

For each unresolved parameter, build a query from the parameter's optimised name + description + the applicable criterion texts. Call `retrieval` with higher top-k (default 12) and hybrid mode if the tool supports it.

### Step 2 ŌĆö Read + update

Same as steps 2ŌĆō4 of the main loop but restricted to the unresolved parameters.

### Step 3 ŌĆö Termination

If a parameter remains unresolved at the end of fallback iteration 3:

* If any `vague` active source was found at any point ŌåÆ Rule-3.
* Else ŌåÆ Rule-4.

Advisory: a parameter finishing at Rule-4 does **not** produce a warning; it is an expected outcome of the silent-all evidence situation.

## Final output

At termination, write:

1. **Per-parameter records** ŌĆö exactly one entry per contract parameter, each validated against the intermediate parameter record schema. Traceability blocks attached per the cardinality rules in the deliverable specification.
2. **Staged `docs_metadata_updates[]`** ŌĆö one patch per touched doc. The patch records which parameters the doc evidenced, and with what classification. The lead applies these via `batch_update_doc_metadata`. The subagent does **not** apply them directly for final output.
3. **Warnings** ŌĆö cross-parameter issues worth surfacing in the category STATE (e.g. "All EV retrieval calls returned the same three docs").
4. **Advisories** ŌĆö `undefined_tier` and `out_of_list` entries.

Assemble these into the result envelope at the tail of the working file, set `"status": "consolidated-ready"`, and return control.

## Invariants

1. Every parameter in the contract has exactly one record in the envelope.
2. Every traceability block references a `doc_id` that appears in `loop_stats.docs_touched`.
3. Every enum-typed field uses a value from the `enums-reference`.
4. Cardinality of traceability blocks matches the cardinality rules (e.g. Rule-4 ŌåÆ zero blocks; Rule-2a/2b ŌåÆ Ōēź2 clear blocks).
5. The subagent never calls any tool outside `tool_allowlist`.
6. The subagent never writes any file other than its own working file.

## Failure handling

| Failure                                     | Response                                                                 |
| :------------------------------------------ | :----------------------------------------------------------------------- |
| A required contract field is missing        | Set envelope `"status": "failed"` with `reason`; return.                 |
| KB retrieval returns nothing, ever          | Produce Rule-4 for every parameter; return `consolidated-ready`.         |
| `ragflow_download` fails for a doc          | Try once more; on repeat failure, skip that doc; note in working file. |
| Cannot build a valid record for a parameter | Produce Rule-4 for that parameter; add a warning; return.                |
