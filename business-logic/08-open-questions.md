# 08 — Open Questions

These are the items that the business description has not yet resolved. Each must be closed before the workflow document is written. They are grouped by area and numbered with a stable prefix so other documents can reference them (e.g. `Q-INPUT-1`).

## Inputs

* **Q-INPUT-1 — Model year.** ✅ Closed. Client must supply model year. No default.
* **Q-INPUT-2 — Market / region.** ✅ Closed. Client must supply market. No default.
* **Q-INPUT-3 — Trim resolution.** ✅ Closed. Trim is fixed to "full option" as a scope rule; client does not specify trim.
* **Q-INPUT-4 — Batch submissions.** ✅ Closed. One car per run, one agent per car. Multiple cars → multiple independent runs, potentially parallel.
* **Q-INPUT-5 — Submission shape.** ✅ Closed 2026-04-22. The skill is format-agnostic at the entry point. It accepts any input format (free-text, JSON, form, conversational message, etc.) and extracts the four required fields (brand, model, year, market) from whatever is provided. If any required field cannot be extracted, the skill prompts for that specific field before proceeding. No fixed schema is enforced on the client. See `11-assumptions.md`.
* **Q-INPUT-6 — Car disambiguation.** ✅ Closed 2026-04-22; confirmed 2026-04-23. When the brand + model + year + market tuple is ambiguous (body-style variants, regional rebadges, sub-models, facelifts, informal names), the agent autonomously selects the most prominent / widespread variant and records the resolved car identity in the deliverable header. The client sees the choice in the output and may request a re-run with an alternative variant if needed. Auto-pick is locked as the default behaviour; no disambiguation gate is added. Risk accepted.
* **Q-INPUT-7 — Market vocabulary.** ✅ Closed 2026-04-22. Hybrid approach: the agent accepts ISO 3166-1 alpha-2 country codes (`DE`, `US`, `AE`), country names (`Germany`, auto-normalized to ISO), and regulatory-zone shortcuts (`EU`, `MEA`, `NAFTA`, `GCC`). Zone shortcuts are resolved to their constituent countries during source discovery. Single-country markets and regulatory zones are both valid inputs; the agent treats them operationally as defined.
* **Q-INPUT-8 — Model-year boundaries.** ✅ Closed 2026-04-22; updated 2026-04-23. No pre-validation of model-year availability. The agent accepts any (model, year) pair without checking whether it is future, discontinued, or valid for the market. The agent proceeds with source discovery and reports whatever is found. If sources do not cover the (model, year) pair, the deliverable will show `Presence = No Information Found, Status = Unable to Decide` records (Rule 4 silent-all). Future and discontinued variants are treated operationally the same as any other model year — sources are the arbiter of coverage.

## Sources

* **Q-SRC-1 — Source scoping.** ✅ Closed. Sources are always per car (see Q-INPUT-4).
* **Q-SRC-2 — Minimum coverage.** ✅ Closed 2026-04-22; updated 2026-04-23. No minimum. Classification starts with whatever sources were approved, even if only one URL. Coverage gaps surface as `Presence = No Information Found, Status = Unable to Decide` (Rule 4) records in the deliverable and can be addressed by the gap-fill workflow if the client chooses to trigger it. See `11-assumptions.md`.
* **Q-SRC-3 — Approval granularity.** ✅ Closed. Approval is **URL-level only**. No site-level concept, no cascade. The gate is mediated by a dedicated *source approval store* (fourth tool capability, planned) with two operations — *flag-for-approval* and *retrieve-approved* — making approval asynchronous and decoupling the validation UI from the harness UI. See `04-sources.md` and `07-tools-and-constraints.md`.
* **Q-SRC-4 — Paywalled content.** ✅ Closed 2026-04-22. Paywalled URLs are silently skipped during source discovery and never proposed to the client. The agent detects the paywall at fetch time and drops the URL from the candidate list. See `11-assumptions.md`.
* **Q-SRC-5 — Language.** ✅ Closed 2026-04-22. All languages are allowed equally. The agent does not filter by language during discovery. Non-English sources can be proposed, approved, and ingested. See `11-assumptions.md`.
* **Q-SRC-6 — Approval completion signal.** ✅ Closed 2026-04-22. Client-initiated kick-off. The agent yields after flagging candidate URLs for approval. The client reviews via the approval store interface and sends an explicit "resume classification" message to the agent. Upon receiving this signal, the agent calls *retrieve-approved* to fetch the final approved set, downloads and uploads them to the KB, then polls the KB ingestion-status check to verify that all uploaded documents have reached the `Ingested` state. Only after all documents are ingested (or a timeout is exceeded) does the agent proceed to the classification stage.
* **Q-SRC-7 — URL deduplication and canonicalisation.** ✅ Closed 2026-04-22. Before calling *flag-for-approval*, the agent canonicalises all URLs to eliminate technical duplicates: (1) lowercase hostname, (2) strip trailing slashes, (3) strip URL fragments, (4) strip common tracking parameters (`utm_*`, `fbclid`, `gclid`, etc.), (5) sort remaining query parameters. After canonicalisation, deduplicate the list — only unique URLs are flagged for approval. This applies to both agent-discovered and client-supplied URLs.
* **Q-SRC-8 — Re-proposing a rejected URL.** ~~Invalid — question dissolved by workflow clarification 2026-04-22.~~ Source discovery and classification are **strictly sequential phases** separated by the client approval gate. Once the gate closes and the client sends the "resume classification" signal, the agent never performs web search again. There is therefore no mechanism by which a rejected URL can be re-encountered during the same run. If the client wants additional sources after the gate, the only two options are: (a) return to the source-discovery phase from the beginning (gate re-opens), or (b) run in automated mode where the agent discovers and ingests without a gate — in which case no URL is "rejected" and re-discovery is handled transparently by URL canonicalisation. AS-SRC-D is retired.
* **Q-SRC-9 — Candidate count cap.** ✅ Closed 2026-04-22. Single global soft cap: **20 URLs** per run in agent-discovered mode, unless the client explicitly sets a different cap at run start. The agent may discover and flag fewer URLs if fewer plausible sources exist for the (brand, model, year, market) tuple. Client-supplied URLs (Mode A) are not counted against the cap. If the client does not specify a cap, 20 is the default.
* **Q-SRC-10 — Retired-state trigger.** ~~Dissolved 2026-04-22 —~~ ~~`Retired`~~ ~~state removed entirely.~~ The `Retired` lifecycle state has been dropped from the source model. When a document needs to be excluded after ingestion (duplicate content, off-target car, content mismatch), it is deleted from the KB directly. No separate `Retired` state or transition rule is needed. AS-SRC-F retired.
* **Q-SRC-11 — Source candidate metadata at flag time.** ✅ Closed 2026-04-22. At *flag-for-approval* time, the agent supplies the following per-URL metadata: (1) **title** (required, falls back to URL host if missing), (2) **excerpt** (required, falls back to empty string if missing), (3) **why selected** (optional — agent's reasoning; omitted if generation is too expensive), (4) **car identity** (required — the canonical brand/model/year/market tuple), (5) **run ID** (required). No candidate is blocked from flagging because metadata is incomplete. See AS-SRC-G.

## Knowledge base

* **Q-KB-1 — Retention.** ✅ Closed 2026-04-22. KB datasets are retained indefinitely after deliverable production. Each KB is scoped to a single run and tagged with the run ID, car tuple, and completion timestamp for audit and traceability. No automatic deletion occurs. The client retains ownership of all KB data.
* **Q-KB-2 — Cross-run reuse.** ✅ Closed 2026-04-22. Each run creates a fresh KB dataset. No reuse of previous ingested documents across runs. Previous datasets are archived but never queried again. Keeps each run fully self-contained and auditable. See `11-assumptions.md`.
* **Q-KB-3 — Progress reporting cadence.** ✅ Closed 2026-04-22. The KB is silently polled for ingestion status without emitting progress updates. A single final "Ingestion complete" message is emitted when all documents reach the `Ingested` state (or timeout). Resume to classification is triggered either by explicit client message or by an automated listener system. See `11-assumptions.md`.
* **Q-KB-4 — Download failure handling.** ✅ Closed 2026-04-22. Approved URLs that fail to download are retried up to 3 times with exponential backoff (1s, 2s, 4s). If still failing, the URL is dropped from the active source set. The failure is recorded (URL, reason, stage: download) and surfaced in the deliverable footer. Classification proceeds with successfully downloaded documents only. The run is never aborted due to a single download failure. See `11-assumptions.md`.
* **Q-KB-5 — Ingestion failure / timeout.** ✅ Closed 2026-04-22. Each downloaded document has a maximum **1-hour (60-minute) ingestion window**. If a document has not reached `Ingested` state after 1 hour (due to parser failure, corrupt file, or slow ingestion), it is dropped from the active source set. The failure is recorded (URL, failure reason: timeout/parse error) and surfaced in the deliverable footer with a note that the document will not be used. Classification proceeds with successfully ingested documents only. See `11-assumptions.md`.
* **Q-KB-6 — Gap-fill KB reuse.** ✅ Closed 2026-04-22. The gap-fill workflow is optional and triggered by the client. It may run with or without the approval gate (client's choice). Gap-fill ingests new approved sources into the **same KB dataset** used by the normal run, not a separate one. New documents are tagged with metadata indicating the cycle number (cycle: 2, cycle: 3, etc.). During retrieval, documents are filtered by cycle number to control which sources are used for which parameters, maintaining clear lineage. See `11-assumptions.md`.
* **Q-KB-7 — Chunk-level citation granularity.** ✅ Closed 2026-04-22. `Source link` remains URL-level only. The canonical URL is stored as metadata on each document when uploaded to the KB. The traceability block includes the URL only; no chunk IDs, fragments, or evidence excerpts. Justification is prose-only, not quote-based. Chunk-level precision is retained internally in the KB for audit but not surfaced in the deliverable. See `11-assumptions.md`.

## Harness

* **Q-HARN-1 — Confidence output.** ✅ Closed. No separate confidence field. Uncertainty is carried by the four-valued `Status` field and by `Presence`.
* **Q-HARN-2 —** **`unknown`** **handling.** ✅ Closed (revised 2026-04-23). The former `unknown` is now `No Information Found` as a **Presence** value only — used exclusively under Rule 4 (silent-all). As a **Status** value it is retired. See Rule 4 in `10-decision-rules.md`. Escalation behaviour handled by the gap-fill workflow (Q-HARN-5).
* **Q-HARN-3-v2 — Conflict resolution priority.** ✅ Closed 2026-04-22. Any disagreement between two or more clear active sources always emits `Status = Conflict` immediately. No source-quality tiebreak is attempted. The client adjudicates all conflicts. See `11-assumptions.md`.
* **Q-HARN-4 — Partial schema coverage.** ✅ Closed. A level whose CSV cell is empty is not a valid target under any circumstance. Evidence of an undefined tier triggers Rule 3 (`Unable to Decide`). See Invariant 7 in `10-decision-rules.md`.
* **Q-HARN-5 — Escalation on silent-all (`No Information Found`** **/ Unable to Decide).** ✅ Closed 2026-04-22; updated 2026-04-23. The **normal pipeline** emits the record as-is (`Presence = No Information Found`, `Status = Unable to Decide`). A **separate gap-fill workflow** (independently triggered) analyses the deliverable for silent-all and weak-coverage parameters and runs a targeted second round of source discovery, approval, ingestion, and classification for those parameters only. The two pipelines are decoupled — the gap-fill is not part of the normal run. See `11-assumptions.md`.
* **Q-HARN-6 — Presence label during Conflict.** ✅ Closed. Split by conflict type. Rule 2a (level conflict): `Presence = Yes`. Rule 2b (presence conflict): `Presence = Disputed` (new value). See `10-decision-rules.md`.
* **Q-HARN-7 — Status for "agreed absent".** ✅ Closed. `Status = Success`, `Presence = No`, `Classification = Empty`.
* **Q-HARN-8 — Presence when Unable to Decide.** ✅ Closed. `Presence = Yes` when vague-only actives mention the parameter. *(See Q-HARN-9 for a related contradiction raised by the client in the same turn.)*
* **Q-HARN-9 — Silent-all outcome:** **`No`** **vs** **`No Information Found`.** ✅ Closed 2026-04-22; **revised 2026-04-23**. When every reviewed source is silent on a parameter, the skill uses Rule 4: `Presence = No Information Found`, `Status = Unable to Decide`, `Classification = Empty`. The 2026-04-22 decision to use `Success/No` is superseded — the distinction between "no evidence found" and "explicit absence" is material and must be visible in the output. See `11-assumptions.md`.
* **Q-HARN-10 — Formal retirement of** **`No Information Found`.** ✅ Closed 2026-04-22; **revised 2026-04-23**. **`No Information Found`** **is reinstated as a valid** **`presence`** **enum value**, used exclusively under Rule 4 (silent-all). It is retired from the **`status`** enum only. Silent-all cases emit `presence = No Information Found`, `status = Unable to Decide`. This must be enforced in all schema and code. See `11-assumptions.md`.
* **Q-HARN-11 — Mixed conflict: valid level vs undefined tier.** ✅ Closed 2026-04-22. **Option A: the undefined-tier source is disregarded for classification purposes.** If Source A supports a valid level (e.g. `High`) and Source B supports an undefined tier (e.g. a `Basic`-like description for an `H`-only parameter), Source B is filtered out. Rule 1 applies on Source A alone: `Presence = Yes`, `Status = Success`, `Classification = High`. **Additionally:** an **intermediate advisory** is recorded for that parameter (a per-parameter artifact separate from the final deliverable) noting that evidence was observed for an undefined tier. The client may use this advisory to identify gaps in the `params.csv` schema and decide whether to add that tier. The advisory is never part of the final deliverable output. See `11-assumptions.md`.
* **Q-HARN-12 — Cross-parameter consistency checks.** ~~Dissolved 2026-04-22 — premise invalid.~~ `artifacts/params.csv` is the authoritative parameter schema and each row is semantically distinct by design. No two parameters refer to the same underlying feature. Superset/subset logical conflicts between parameters cannot arise because the CSV prohibits duplicate or overlapping parameter definitions. No cross-parameter consistency check is needed. AS-HARN-C retired.
* **Q-HARN-13 — Parameter partition strategy.** ✅ Closed 2026-04-22; threshold finalised 2026-04-23. **Option C: partition by category, with large categories split into sub-groups.** Partitions are formed by category (EV, INFOT., REMOTE APP, ADAS Driving, ADAS Danger Alert, ADAS Parking, ADAS Trailering, Cockpit, Trail & Truck Specific). **Hard cap: max 15 parameters per partition.** Any category with more than 15 parameters is sub-divided into semantically coherent sub-groups of ≤ 15 parameters each (e.g. Cockpit → Cockpit-Display, Cockpit-Audio, Cockpit-Seating). The partition is computed at run-start from the live `params.csv` row count per category. See `11-assumptions.md`.
* **Q-HARN-14 — Determinism / reproducibility.** ✅ Closed 2026-04-22. Non-determinism is accepted and disclosed. Two runs on the same car, same CSV, same approved source set are not guaranteed to emit identical deliverables. The deliverable header explicitly states this. The client is told explicitly: re-runs are requested for coverage reasons (new sources), not reproducibility. See `11-assumptions.md`.

## Deliverable

* **Q-DEL-1 — Serialisation format.** ✅ Closed 2026-04-22. **Option D: JSON primary + CSV secondary.** The primary deliverable is a structured JSON file (`<brand>-<model>-<year>-<market>-<run-id>.json`) for machine integration and downstream systems. A secondary CSV export is generated alongside the JSON, flattening the per-parameter records into tabular form for client review in spreadsheet tools. Traceability blocks in JSON are preserved in full; CSV representation handles nesting via concatenation or multiple rows as needed. See `11-assumptions.md`.
* **Q-DEL-2 — Review pass.** ✅ Closed 2026-04-22. No intermediate review gate. The merged deliverable is emitted directly as final output. The client reviews it post-delivery and triggers the gap-fill workflow or a re-run if they disagree with any classification. See `11-assumptions.md`.
* **Q-DEL-3 — Re-run / revision.** ✅ Closed 2026-04-22. **Flexible via workflows.** Default re-run covers the full car (full source re-ingestion and re-classification). Workflows support two additional re-run patterns: (1) **per-parameter flagging** — client marks specific parameters for review; (2) **gap-driven re-run cycles** — when many parameters are `No Information Found`, client can trigger a targeted re-run cycle focusing on those parameters, optionally with or without the approval gate (client's choice). All three patterns are workflow-level concerns; the core business logic supports full-car and provides parameter-specific re-classification. Workflow definitions for per-parameter flagging and gap-driven cycles are deferred to the workflow design phase. See `11-assumptions.md` and Q-DEL-10.
* **Q-DEL-4 — Schema type encoding.** ✅ Closed 2026-04-22. **Compact notation.** The `Schema type` field uses abbreviated form: `"B/H"` (Basic+High levels allowed), `"B/M/H"` (all three levels), `"H"` (High only), etc. Compact and familiar in CSV contexts. A legend explaining the abbreviations is included in the deliverable header or schema documentation. See `11-assumptions.md`.
* **Q-DEL-5 — Run-level summary.** ✅ Closed 2026-04-22; updated 2026-04-23. **Option D: Summary + extended analytics.** The deliverable includes a run-level summary with counts (Total Parameters, Yes, No, No Information Found, Conflict, Unable to Decide), distribution by schema type (Basic, Medium, High), and lists of all Conflict/Unable-to-Decide/No-Information-Found parameters. Extended telemetry section includes: source coverage (approved/ingested/failed counts), ingestion time, classification time, classification execution log excerpt. See `11-assumptions.md`.
* **Q-DEL-6 — Run-header metadata.** ✅ Closed 2026-04-22. **Option C: Extended header.** The deliverable header includes: run ID, submission timestamp, completion timestamp, resolved car identity (brand, model, year, market), resolved full-option trim name, CSV snapshot version/hash, counts (approved/rejected/ingested/failed sources), **ingestion stage start/end times, classification stage start/end times, total elapsed time (HH:MM:SS), classification partition count.** All telemetry in a single self-contained header. See `11-assumptions.md`.
* **Q-DEL-7 — Record ordering.** ✅ Closed 2026-04-22. **Option A: CSV row order.** Per-parameter records follow the exact order of `artifacts/params.csv`, preserving category grouping and author intent. Applies to JSON array ordering and CSV row sequence. See `11-assumptions.md` AS-DEL-B.
* **Q-DEL-8 — Failure reporting in the deliverable.** ✅ Closed 2026-04-22. **Option B: Footer section only.** A dedicated `sources_excluded` section appears in the deliverable footer listing all sources that failed to download or ingest. Columns: canonical URL, stage of failure (download / ingestion), short reason (e.g. "HTTP 403 Forbidden", "parse timeout"). No per-parameter notes. Rationale: failed sources were never ingested, so we have no data about whether they would have been relevant to any specific parameter. Footer provides audit visibility without false per-parameter associations. See `11-assumptions.md` AS-DEL-C.
* **Q-DEL-9 — Deliverable persistence.** ✅ Closed 2026-04-22. **Run-workspace model with internal-workspace folder.** Each run owns a dedicated workspace directory. Inside it: (1) the final deliverable files (JSON + CSV) at the workspace root; (2) an internal-workspace subfolder containing run-state, per-category consolidated records, per-partition working files, advisories, and source artefacts. The internal workspace is the run's persistent memory — enabling resumability, gap identification, and post-run analysis. It is not part of the client-facing deliverable but is retained alongside it. The exact layout, file names, and writer roles are framework-implementation concerns; see the framework documentation for details. See `11-assumptions.md` AS-DEL-D.
* **Q-DEL-10 — Re-run vs gap-fill reconciliation.** ✅ Closed 2026-04-22. No contradiction. **Three distinct re-run workflows:**
  1. **Full-car re-run** (default): entire source re-approval/ingestion + full parameter re-classification. Creates a new run ID. New run is independent of the original.
  2. **Per-parameter flagging** (workflow-driven): client marks specific parameters from the existing deliverable for review. Workflow logic determines whether this is a targeted single-parameter re-classification on the same KB or a targeted gap-fill mini-cycle. Handled separately from the normal pipeline.
  3. **Gap-fill workflow** (optional, coverage-driven): client reviews deliverable, identifies silent-all/weak-coverage parameters, and initiates gap-fill. Gap-fill discovers/approves new sources (optionally with gate; client chooses), ingests to the same KB with cycle metadata, and re-classifies only the targeted parameters. Results merge with original deliverable.
  * Full-car re-runs are the default, straightforward path.
  * Gap-fill and per-parameter flagging are optional workflows, decoupled from the normal pipeline.
  * Q-DEL-3 (re-run granularity) and Q-HARN-5 (gap-fill) are complementary, not redundant. See `11-assumptions.md` AS-DEL-E.

## Scope

* **Q-SCOPE-1 — Definition of "full option".** ✅ Closed 2026-04-22. **Option A: Manufacturer's published top trim.** The agent identifies the highest trim level published by the manufacturer for the specified model/year/market and treats it as the scope target. The resolved trim name is recorded in the deliverable header. If multiple top trims exist (e.g., Sport vs Luxury premium variants), the agent selects the most feature-rich one. See `11-assumptions.md` AS-SCOPE-1.
* **Q-SCOPE-2 — CSV evolution.** ✅ Closed 2026-04-22. **Frozen per run; developer-owned; injected into workspace.** `artifacts/params.csv` is owned by developers (not the client). It is injected into the agent's run workspace at the start of each run and frozen for the entirety of that run — including all gap-fill cycles within the same run. A new CSV version may only take effect in a completely new, independent run. No mid-run or mid-cycle CSV changes are permitted. See `11-assumptions.md` AS-SCOPE-2.
* **Q-SCOPE-3 — Concurrent runs on the same car.** ✅ Closed 2026-04-22. **Allowed; fully independent.** Two or more runs targeting the same (brand, model, year, market) tuple may run simultaneously. Each run is treated as entirely unrelated to any other — it has its own run ID, workspace, knowledge base, approval store, source set, classification pass, and deliverable. No coordination, de-duplication, or merging occurs between concurrent runs. See `11-assumptions.md` AS-SCOPE-A.
* **Q-SCOPE-4 — Scope of "reviewed sources".** ✅ Closed 2026-04-22. **Option B: Approved + ingested + actually queried (per-parameter).** "Reviewed" means only the documents that retrieval actually returned above the relevance threshold for a specific parameter. The denominator is per-parameter, not global. Each classification partition records in its intermediate working file which source documents were returned by retrieval for each parameter it processed. The records are aggregated into metadata about source coverage across the full run. See `11-assumptions.md` AS-HARN-F.
* **Q-SCOPE-5 — Out-of-list findings.** ✅ Closed 2026-04-22. **Option C: Surfaced internally only.** Features found in sources but absent from `artifacts/params.csv` are recorded in the partition's internal working file (not in the client-facing deliverable). Developers can inspect these findings to identify CSV gaps and evolve the parameter list in future versions. The client-facing deliverable remains clean and scoped strictly to the CSV. See `11-assumptions.md` AS-SCOPE-B.
