# 11 — Assumptions

This document collects every assumption made during the design of this skill — both **settled assumptions** (decisions that have been explicitly agreed) and **working assumptions** (placeholders adopted to keep progress moving, not yet confirmed by the client). Each entry records what is assumed, why, and where it was decided.

## Architectural assumptions

These are not business-logic decisions but structural commitments about how the skill is built. They constrain the workflow design.

|  ID                                                                 | Assumption                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                          | Rationale                                                                                                                                                                                                                            |
| :------------------------------------------------------------------ | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ARCH-1 — Platform agnosticism**                                   | The skill makes no assumption about which agent platform runs it. Any system that can: (1) spawn child agents, (2) access the four tool capability categories (search, browser/scraper, KB, approval store), and (3) read and write files — can execute this skill. No Claude-specific, vendor-specific, or framework-specific primitives are used in the business logic or workflow design.                                                                                                                                        | Decided 2026-04-22. Keeps the skill portable and future-proof. Concrete platform bindings live only in the implementation layer, not in the business logic or workflow documents.                                                    |
| **ARCH-2 — Parallel classification via parameter partitions**       | After ingestion is complete, the parameter list is partitioned and each partition is classified in parallel. Each partition independently produces Presence / Status / Classification / Confidence / traceability blocks for its subset. Partition outputs are then consolidated into the final deliverable. The exact partition strategy (by category, by count, by other heuristic) is a framework concern.                                                                                                                       | Decided 2026-04-22. Parallelism is the primary performance lever for the \~160-parameter classification pass. Partitions are stateless with respect to each other — each queries the same shared KB but makes independent decisions. |
| **ARCH-3 — High-level normal pipeline stage order**                 | The normal pipeline follows this stage order: (1) acquire source candidates, (2) wait for client approval via the approval store, (3) download and ingest approved sources, (4) wait for full ingestion, (5) classify parameter partitions in parallel, (6) consolidate results and emit deliverable. Details of each stage are deferred to the framework documentation.                                                                                                                                                            | Decided 2026-04-22. Recorded here as an orientation anchor; the framework documentation expands each stage.                                                                                                                          |
| **ARCH-4 — Hard phase separation: search before gate, never after** | Source discovery and classification are strictly sequential and non-overlapping. The agent performs all web searching and URL flagging before the approval gate. Once the client sends the "resume classification" signal, the agent never performs web search again within that run. If the client wants additional sources post-gate, the only paths are: (a) restart the source-discovery phase (gate re-opens), or (b) automated mode — agent discovers and ingests without a gate. No hybrid mid-run re-discovery is possible. | Decided 2026-04-22. Eliminates the ambiguity of Q-SRC-8 entirely. Keeps the pipeline deterministic: once classification starts, the source set is fixed.                                                                             |

These have **not** yet been confirmed by the client. They are placeholders that let the design move forward. Each must be closed before the workflow document is finalised.
